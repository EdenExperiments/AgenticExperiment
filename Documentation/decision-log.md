| 2026-03-15 | D-011 | AI-assisted skill calibration in the onboarding flow is optional, not mandatory. A manual starting-level selection must always be available as a fallback path. | Mandatory AI calibration creates a hard onboarding dependency on the user's Claude API key being valid at signup time. The AI calibration path remains the recommended default but must never be the only path. |
| 2026-03-15 | D-012 | Release 1 uses email/password auth only. Social auth (OAuth providers) is deferred to a later release. | Keeps the auth schema minimal for release 1 and avoids OAuth provider integration scope. Social auth can be layered on after the core loop is stable. |
| 2026-03-15 | D-013 | The XP progression curve is non-linear. The XP required to advance to each successive level increases, so higher levels demand progressively more effort. The exact curve shape (e.g., quadratic, exponential, custom step function) is left to the architecture-agent to propose, but the growth direction is confirmed. | Reflects that real-world skill mastery becomes harder as you progress; non-linear growth is intentional. **Resolved by D-014.** |
| 2026-03-15 | D-014 | XP curve shape confirmed: quadratic with tier-based base multipliers. Formula: `XP_to_reach_level(N) = base_multiplier(tier(N)) × N²`. **Tier structure (revised 2026-03-15 per user direction — Master now starts at level 100):** Levels 1–9 = 100 (Novice), Levels 10–19 = 120 (Apprentice), Levels 20–29 = 150 (Journeyman), Levels 30–59 = 200 (Expert), Levels 60–99 = 260 (Veteran), Levels 100+ = 350 (Master). Representative thresholds: L1=100, L5=2,500, L9=8,100, L10=12,000, L20=60,000, L29=126,150, L30=180,000, L60=936,000, L100=3,500,000, L150=7,875,000, L200=14,000,000. MaxLevel = 200. Tier transitions at levels 10, 20, and 30 correspond to blocker gate levels (9, 19, 29) and remain unchanged. The blocker gates at 9, 19, and 29 are confirmed product decisions and are not affected by this revision. The level computation is a pure function in the `xpcurve` Go package. See architecture.md section 0 and 2 for the full formula and Go code. | Resolves D-013. Revised post-requirements-pass per user direction: Master tier starts at level 100, not level 50. A Veteran tier (levels 60–99) is added to bridge Expert and Master. This makes Master an elite long-term aspiration requiring years of consistent daily use. MaxLevel is raised to 200 to give Master-tier users continued progression headroom. The three blocker gates (levels 9, 19, 29) are unaffected. Quadratic curve with tier multipliers is easy to explain to users, produces clean numbers, avoids floating-point complexity, and aligns tier transitions with the existing blocker gate levels. Phase 2 schema work (step 2a) is now unblocked. |
| 2026-03-15 | D-015 | Claude API key encryption confirmed as AES-256-GCM envelope encryption at the Go application layer. Supersedes A-001. The approach is confirmed (not just assumed) with the following production requirement: in production, the master key must be loaded from a secrets manager at startup (Kubernetes secret, AWS Secrets Manager, GCP Secret Manager, or HashiCorp Vault), not from a bare environment variable. In development, an environment variable is acceptable. All other details from A-001 are confirmed unchanged: per-user DEK wrapped by master key, decryption only in the Go process at request time, key never logged or returned to client, validation at save time, rotation via DEK re-encryption. Supabase Vault was evaluated and rejected due to infrastructure coupling risk. A Go KMS library was evaluated and rejected as excess operational complexity at this stage. | Supabase Vault tightly couples key management to Supabase as an infrastructure dependency; if the project migrates providers, vault-encrypted data requires a migration path. The Go-layer AES-256-GCM approach satisfies D-009, is fully testable, keeps the decryption surface narrow (single Go process), and is portable. The production secrets-manager requirement hardens A-001's weakest point. F-003 build is now unblocked. |
| 2026-03-15 | D-016 | Master tier starts at level 100. Reaching Master requires years of consistent daily use and is intended as an elite aspirational milestone. Only the most disciplined users are expected to reach it. MaxLevel is set to 200 to give Master-tier users continued progression headroom. The top 100 levels (100–200) are all Master tier — there is no further tier graduation beyond Master, only continuous hard grinding. | Reflects the design intent that Master should be extraordinary rather than reachable within a typical engagement window. Setting MaxLevel to 200 ensures that even users who do reach Master have meaningful continued progression rather than hitting a hard cap. |

## Implementation Risks (added by architecture-agent)

| Date | ID | Risk | Mitigation |
| --- | --- | --- | --- |
| 2026-03-15 | R-001 | Supabase JWKS cache staleness: if the Go process caches the Supabase JWKS and Supabase rotates its signing keys, JWT validation will fail for all users until the cache is refreshed. | Cache JWKS with a 1-hour TTL. On JWT validation failure due to unknown key ID, re-fetch JWKS once before rejecting the token. |
| 2026-03-15 | R-002 | AES-256-GCM nonce reuse: catastrophic if a nonce is reused with the same key — ciphertext confidentiality is fully broken. | Always use `crypto/rand` to generate a fresh 12-byte nonce per encryption operation. Never derive nonces deterministically or from counters. Enforced by code review and unit tests. |
| 2026-03-15 | R-003 | XP aggregate drift: `skills.current_xp` is a denormalised aggregate; if a write to `skills` succeeds but the `xp_events` write fails (or vice versa), they drift silently. `skills.current_level` adds a third value that must stay in sync: every `xp_events` insert requires a three-way atomic update — `xp_events` insert → `skills.current_xp` update → `skills.current_level` update (via `LevelForXP(current_xp)`) — all within a single `BEGIN`/`COMMIT` transaction. | Always update all three in a single database transaction. Provide a reconciliation query: `UPDATE skills SET current_xp = (SELECT COALESCE(SUM(xp_delta), 0) FROM xp_events WHERE skill_id = skills.id)` for operational use; also recompute `current_level` from the reconciled `current_xp` in the same statement. |
| 2026-03-15 | R-004 | Level gate bypass: if effective-level computation lives only in the template layer, a user crafting direct API requests could observe an uncapped level or trigger actions that assume a higher level. | The `effective_level()` computation must live in the Go service/handler layer. Templates receive the already-capped level value and have no access to the raw level. |
| 2026-03-15 | R-005 | `LevelForXP` infinite loop: without an upper bound, the loop in `LevelForXP` iterates until `XPToReachLevel(level+1)` exceeds `totalXP`, which is guaranteed by the formula but not by any explicit guard — a bug in `XPToReachLevel` could cause an infinite loop. | Mitigated by the `MaxLevel = 200` constant in the `xpcurve` package (raised from 99 per D-016); the loop breaks at `MaxLevel` regardless of XP value. |

| 2026-03-15 | D-017 | Mobile navigation is a fixed bottom tab bar with four items: Dashboard, LifeQuest, NutriLog (Coming Soon placeholder), Account. Hamburger menus and top tab bars are rejected. The NutriLog slot is present from release 1 so the nav layout is stable at NutriLog launch. The bottom tab bar is hidden during multi-step flows (skill creation); a back chevron replaces it. Desktop uses a left sidebar (collapsible on medium viewports). | Bottom tab bars give natural thumb reach and do not require an extra tap to reveal the menu. Four items is within standard guidelines. Including NutriLog as a placeholder from day one prevents a navigation restructure when NutriLog launches. See ux-spec.md Section 2. |
| 2026-03-15 | D-018 | Master tier (levels 100–200) is excluded from the starting-level picker during skill creation. The maximum self-reported starting level is 99 (top of Veteran tier). A user who is genuinely Master-caliber starts at a high Veteran level and advances quickly via normal XP accumulation. The Go handler for skill creation must reject starting_level values > 99. | Allowing self-reported Master starting levels would undermine Master's aspirational meaning as an elite multi-year achievement (D-016). The skill creation handler must enforce this as a server-side validation, not just a UI constraint. See ux-spec.md Section 3.2. |
| 2026-03-15 | D-019 | Quick XP logging uses a bottom sheet (mobile) or modal dialog (desktop). It does not navigate to a new route. The canonical three-tap sequence is: (1) tap skill card, (2) tap XP amount chip, (3) tap Log XP button. Preset XP amounts are 50, 100, 250, 500. A "Custom" chip allows free-form numeric input. The last-used XP amount is pre-selected on return visits using session or local state (never server-persisted). | A bottom sheet avoids a full navigation change, keeps the user on the skill list, and enables logging XP for multiple skills in sequence. The three-tap path with preset amounts satisfies the Phase 2 exit criterion of three taps or fewer on mobile. Custom amount requires an additional tap (acceptable as a fourth tap for the minority case). See ux-spec.md Section 4. |
| 2026-03-15 | D-020 | A six-tier color system is confirmed for XP progress bars, tier badges, and skill card accents. Novice: Gray-400; Apprentice: Blue-500; Journeyman: Green-500; Expert: Purple-600; Veteran: Amber-600; Master: Gold/Yellow-500 with gradient fill. Master uses gold/amber to visually distinguish it from all other tiers as the exceptional tier. This color system is binding for all implementation. | Tier colors make progression state immediately scannable at a glance and reinforce the emotional meaning of each tier. Master's gold treatment is a deliberate design signal — it is not merely the "next color in the sequence." See ux-spec.md Section 5.3. |
| 2026-03-15 | D-021 | When a blocker gate is active, the gate section replaces the XP progress bar on the skill detail screen (it does not appear below it). The gate section occupies the same vertical position as the bar, keeping gate state above the fold on mobile. The gate section displays: lock icon, gate title, gate description, current XP accruing, display level (capped), raw XP-computed level, and an explanatory line that completion is coming in a future update. | If the gate section were appended below the XP bar, the most important state information (progression is locked) would be hidden below less-important content on a small screen. Gate-active state must be the dominant visual element on the skill detail screen when it is in effect. See ux-spec.md Section 6. |
| 2026-03-15 | D-022 | Tier transitions (e.g., Novice → Apprentice at level 10) surface as a full-screen overlay modal, not a toast. The modal includes: new tier name, tier description, tier badge, XP jump explainer ("the next tier requires more XP per level — this reflects that advanced mastery takes greater effort"), and a full-width dismiss button. Routine level-ups within a tier use a toast only. | A toast is appropriate for routine events. A tier transition is a milestone that benefits from a dedicated moment and is the correct place to address the known UX risk that tier-boundary XP gaps may confuse users. The XP jump explainer embedded in the modal preempts confusion about whether the curve is a bug. See ux-spec.md Section 5.4 and architecture.md section 0 (known UX risk). |
