## Implementation Risks (added by architecture-agent)

| Date | ID | Risk | Mitigation |
| --- | --- | --- | --- |
| 2026-03-15 | R-001 | Supabase JWKS cache staleness: if the Go process caches the Supabase JWKS and Supabase rotates its signing keys, JWT validation will fail for all users until the cache is refreshed. | Cache JWKS with a 1-hour TTL. On JWT validation failure due to unknown key ID, re-fetch JWKS once before rejecting the token. |
| 2026-03-15 | R-002 | AES-256-GCM nonce reuse: catastrophic if a nonce is reused with the same key — ciphertext confidentiality is fully broken. | Always use `crypto/rand` to generate a fresh 12-byte nonce per encryption operation. Never derive nonces deterministically or from counters. Enforced by code review and unit tests. |
| 2026-03-15 | R-003 | XP aggregate drift: `skills.current_xp` is a denormalised aggregate; if a write to `skills` succeeds but the `xp_events` write fails (or vice versa), they drift silently. `skills.current_level` adds a third value that must stay in sync: every `xp_events` insert requires a three-way atomic update — `xp_events` insert → `skills.current_xp` update → `skills.current_level` update (via `LevelForXP(current_xp)`) — all within a single `BEGIN`/`COMMIT` transaction. | Always update all three in a single database transaction. Provide a reconciliation query: `UPDATE skills SET current_xp = (SELECT COALESCE(SUM(xp_delta), 0) FROM xp_events WHERE skill_id = skills.id)` for operational use; also recompute `current_level` from the reconciled `current_xp` in the same statement. |
| 2026-03-15 | R-004 | Level gate bypass: if effective-level computation lives only in the frontend, a user crafting direct API requests could observe an uncapped level or trigger actions that assume a higher level. | The `effective_level()` computation must live in the Go service/handler layer. The API response includes the already-capped level value; the frontend has no access to the raw level. |
| 2026-03-15 | R-005 | `LevelForXP` infinite loop: without an upper bound, the loop in `LevelForXP` iterates until `XPToReachLevel(level+1)` exceeds `totalXP`, which is guaranteed by the formula but not by any explicit guard — a bug in `XPToReachLevel` could cause an infinite loop. | Mitigated by the `MaxLevel = 200` constant in the `xpcurve` package (raised from 99 per D-016); the loop breaks at `MaxLevel` regardless of XP value. |

| 2026-03-15 | D-017 | Mobile navigation is a fixed bottom tab bar with four items: Dashboard, LifeQuest, NutriLog (Coming Soon placeholder), Account. Hamburger menus and top tab bars are rejected. The NutriLog slot is present from release 1 so the nav layout is stable at NutriLog launch. The bottom tab bar is hidden during multi-step flows (skill creation); a back chevron replaces it. Desktop uses a left sidebar (collapsible on medium viewports). | Bottom tab bars give natural thumb reach and do not require an extra tap to reveal the menu. Four items is within standard guidelines. Including NutriLog as a placeholder from day one prevents a navigation restructure when NutriLog launches. See ux-spec.md Section 2. |
| 2026-03-16 | D-018 | Legend tier (levels 100–200) is excluded from the starting-level picker during skill creation. The maximum self-reported starting level is 99 (top of Grandmaster tier). A user who is genuinely Legend-caliber starts at a high Grandmaster level and advances quickly via normal XP accumulation. The Go handler for skill creation must reject starting_level values > 99. | Allowing self-reported Legend starting levels would undermine Legend's aspirational meaning as an elite multi-year achievement (D-016). The skill creation handler must enforce this as a server-side validation, not just a UI constraint. See ux-spec.md Section 3.2. |
| 2026-03-15 | D-019 | Quick XP logging uses a bottom sheet (mobile) or modal dialog (desktop). It does not navigate to a new route. The canonical three-tap sequence is: (1) tap skill card, (2) tap XP amount chip, (3) tap Log XP button. Preset XP amounts are 50, 100, 250, 500. A "Custom" chip allows free-form numeric input. The last-used XP amount is pre-selected on return visits using session or local state (never server-persisted). | A bottom sheet avoids a full navigation change, keeps the user on the skill list, and enables logging XP for multiple skills in sequence. The three-tap path with preset amounts satisfies the Phase 2 exit criterion of three taps or fewer on mobile. Custom amount requires an additional tap (acceptable as a fourth tap for the minority case). See ux-spec.md Section 4. |
| 2026-03-16 | D-020 | An eleven-tier color system is confirmed for XP progress bars, tier badges, and skill card accents. Color progression moves cold → warm → gold, communicating increasing prestige. Novice: Gray-400; Apprentice: Blue-500; Adept: Teal-500; Journeyman: Green-500; Practitioner: Lime-500; Expert: Purple-600; Veteran: Fuchsia-600; Elite: Amber-600; Master: Orange-600; Grandmaster: Red-600; Legend: Yellow-500 with gradient fill. Legend retains the gold/gradient treatment as the exceptional top tier. This color system is binding for all implementation. | Tier colors make progression state immediately scannable at a glance and reinforce the emotional meaning of each tier. The cold-to-warm-to-gold arc creates an intuitive visual hierarchy. Legend's gold treatment is a deliberate design signal — it is not merely the "next color in the sequence." Updated 2026-03-16 from 6 tiers to 11 tiers. See ux-spec.md Section 5.3. |
| 2026-03-15 | D-021 | When a blocker gate is active, the gate section replaces the XP progress bar on the skill detail screen (it does not appear below it). The gate section occupies the same vertical position as the bar, keeping gate state above the fold on mobile. The gate section displays: lock icon, gate title, gate description, current XP accruing, display level (capped), raw XP-computed level, and an explanatory line that completion is coming in a future update. | If the gate section were appended below the XP bar, the most important state information (progression is locked) would be hidden below less-important content on a small screen. Gate-active state must be the dominant visual element on the skill detail screen when it is in effect. See ux-spec.md Section 6. |
| 2026-03-15 | D-022 | Tier transitions (e.g., Novice → Apprentice at level 10) surface as a full-screen overlay modal, not a toast. The modal includes: new tier name, tier description, tier badge, XP jump explainer ("the next tier requires more XP per level — this reflects that advanced mastery takes greater effort"), and a full-width dismiss button. Routine level-ups within a tier use a toast only. | A toast is appropriate for routine events. A tier transition is a milestone that benefits from a dedicated moment and is the correct place to address the known UX risk that tier-boundary XP gaps may confuse users. The XP jump explainer embedded in the modal preempts confusion about whether the curve is a bug. See ux-spec.md Section 5.4 and architecture.md section 0 (known UX risk). |

| 2026-03-21 | D-033 | **Gate auto-clear on high starting level (revised 2026-03-23).** When a skill is created with a starting level that crosses gate boundaries, all gates at or below the starting level are auto-cleared at creation time (`blocker_gates.is_cleared = true`, `gate_submissions` row inserted with `verdict = 'self_reported'`, `evidence_*` fields set to a system-generated note). The first gate above the starting level is the next challenge. Example: starting at level 28 → gates 9 and 19 auto-cleared, gate 29 is next. Enforced inside `CreateSkill()` in the same transaction as skill/gate insertion. `EffectiveLevel()` unchanged. | Original design required the highest boundary gate below starting level, causing confusing UX: Arbiter suggests 28, user sees 19 as their displayed level. Revised so that effective_level matches starting_level — the next challenge is the gate ahead, not behind. |

| 2026-03-21 | D-034 | **Quick Log switches to time-primary input. Supersedes D-019 XP chip presets.** Users select time spent (preset chips: 15, 30, 45, 60 min, plus Custom up to 480 min). XP is derived server-side via: `XP = minutes × 3 × (1 + 0.4 × (tierNum − 1))`, rounded to nearest integer. The `≈ N XP` preview is shown beneath the selected chip in muted text. PostSessionScreen is unaffected — session time is auto-tracked, XP selection there reflects effort quality, not time input. Quick Log `onSubmit` continues to send `xpDelta` (computed client-side from time × rate) and `timeSpentMinutes`. The base rate of 3 XP/min at tier 1 is a binding constant. At tier 1: 15 min→45 XP, 30 min→90 XP, 45 min→135 XP, 60 min→180 XP. | Inputting XP directly invites inflation and feels arbitrary. Time is a grounded, honest input that is harder to game psychologically. Consistent with session-based flow which already converts time to XP automatically. Recommended by UX agent 2026-03-21. |

| 2026-03-23 | D-035 | **Three-theme system.** The app offers three switchable UI themes: Minimal (light, data-forward, productivity), Retro (dark, amber/gold, pixel art, RPG narrative), Modern (dark navy, cyan/magenta, glass morphism, neon). All themes surface identical features — differences are visual treatment and UX flavour only. Architecture: three-layer (CSS custom properties ~60%, theme-scoped component CSS ~25%, component variants with code splitting ~15%). See product-requirements.md and PLATFORM-DECISIONS.md for full details. | Users have different aesthetic preferences. Offering three themes caters to productivity-focused users (Minimal), RPG fans (Retro), and users who want atmospheric polish (Modern). The three-layer architecture keeps ~85% of theming in CSS with zero JS overhead, only creating component variants for structurally different elements. |
| 2026-03-23 | D-036 | **Development pipeline split: TDD for logic, reviewer for UI.** Logic/API work follows spec → TDD → implement → code review. UI/visual work follows style guide → page brief → implement → visual review (no TDD gate). The reviewer agent gains a Visual Review mode for UI work, checking design token usage, theme compatibility, accessibility, and three-layer architecture compliance. | TDD verifies behaviour, not aesthetics. Applying TDD universally to visual work forces meaningless assertions about CSS classes, causes agent friction, and constrains creative freedom. Logic/API tests remain mandatory for business rules, API contracts, and data flow. |
| 2026-03-23 | D-037 | **Hub architecture.** The RPG Tracker (LifeQuest) is the central hub for the suite. NutriLog, MindTrack, and future apps feed progress into the unified character/skill progression system. All suite apps contribute to overall character development — the hub is not just an aggregation dashboard. | The product mission is "levelling up as a person" across all domains. A hub model where every app feeds into unified progression creates a cohesive experience and gives users a single view of their growth. |

| 2026-03-23 | D-041 | **Primary skill pin: single pin only.** Each user has at most one pinned "primary skill" stored as `primary_skill_id` on the users table (nullable FK). When no skill is pinned, the dashboard algorithmically selects one based on streak length, favourite status, and recency. Multi-pin is not needed — the existing favourites system (F-033) serves that role. | The dashboard page guide calls for a single centre-stage "Next Quest" card. A single pin gives one clear answer to "what should I work on?" Multiple pins would dilute focus and duplicate the favourites feature. |

| 2026-03-23 | D-042 | **Avatar storage: Supabase Storage.** User avatars stored in a public Supabase Storage bucket `avatars` at deterministic path `{user_id}/avatar`. Go API accesses storage via direct REST API (HTTP PUT/DELETE with service role key) — no Go SDK. `avatar_url` column on `users` table (nullable TEXT). Client-side crop to 256x256 JPEG before upload. Max 2MB. | Zero new providers — Supabase Storage is already part of the platform. RLS policies for per-user access. Direct REST calls avoid adding an immature Go SDK dependency and match the project's existing pattern of minimal external dependencies. |
